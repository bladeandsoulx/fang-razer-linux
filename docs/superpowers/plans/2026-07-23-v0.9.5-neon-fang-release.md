# Fang v0.9.5 Neon Fang Release Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Publish Fang v0.9.5 as an immutable six-asset release whose one-command installer serves the Neon Fang banner, synchronized beginner documentation, and matching 0.9.5 packages.

**Architecture:** Keep the existing installer, package builders, release-contract staging code, and publish-once GitHub workflow unchanged. Add release-specific documentation contracts, synchronize every governed version source, rebuild and stage all packages locally as a preflight, then tag only the exact pushed commit whose full `main` CI run succeeds. The tag workflow independently rebuilds, tests, stages, and publishes the official assets before release metadata and live download behavior are verified.

**Tech Stack:** Bash, Node.js 22 test runner, Svelte 5, Rust/Cargo, Tauri 2, cargo-deb, RPM tooling, ShellCheck, Git, GitHub Actions, GitHub CLI, GitHub REST API

## Global Constraints

- Version is exactly `0.9.5`; tag is exactly `v0.9.5`.
- Release title is exactly `Fang v0.9.5 — Neon Fang installer`.
- Comparison range is exactly `v0.9.4...v0.9.5`.
- This release contains the Neon Fang banner, beginner README improvements, and release-documentation safety checks only.
- Do not claim new application behavior, hardware support, Linux distributions, base versions, or architectures.
- Keep `v0.9.4`, its tag, metadata, and six immutable assets unchanged.
- Keep the public installer URL on `releases/latest`; never point it to raw `main`.
- Build and publish exactly `install.sh`, `SHA256SUMS`, two DEBs, and two RPMs.
- Official release assets come only from the tag-triggered GitHub workflow; local desktop artifacts are preflight evidence and are never uploaded manually.
- Do not create or push `v0.9.5` until the exact pushed `main` commit has a successful CI run, the tag/release absence checks pass, immutable releases are enabled, and the temporary credential is ready.
- Never print, log, or commit the value used for `IMMUTABLE_RELEASES_TOKEN`.
- Delete `IMMUTABLE_RELEASES_TOKEN` after the published release and live installer pass verification.
- If a release draft exists after failure, inspect it before any retry; never blindly rerun, delete a published immutable release, replace assets, or reuse a tag for different source.
