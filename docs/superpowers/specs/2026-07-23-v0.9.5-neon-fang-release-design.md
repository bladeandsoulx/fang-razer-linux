# Fang v0.9.5 Neon Fang release

**Date:** 2026-07-23
**Status:** Approved for planning

## Goal

Publish Fang `v0.9.5` as a focused, immutable six-asset release so the
documented one-command installer serves the Neon Fang banner already present on
`main`.

The release also carries the beginner-friendly README and release-documentation
safety checks added after `v0.9.4`. It does not introduce or claim new
application behavior, hardware support, or installer platform support.

## Current-state constraint and root cause

The public installer command is:

```sh
curl -fsSL https://github.com/bladeandsoulx/fang-razer-linux/releases/latest/download/install.sh | bash
```

GitHub resolves that URL to the `install.sh` asset attached to the latest
published release. The current latest release is the immutable `v0.9.4`
release, whose installer contains the original box banner. Pushing the Neon
Fang implementation to `main` therefore cannot change the command's output.

`v0.9.4` must remain unchanged. Its tag, release metadata, and six attached
assets must not be deleted, replaced, or republished. `v0.9.5` is the first
release whose attached installer contains the Neon Fang banner.

## Release identity and scope

- Version: `0.9.5`
- Tag: `v0.9.5`
- Title: `Fang v0.9.5 — Neon Fang installer`
- Release type: stable, published, non-prerelease, immutable, and marked latest
- Comparison range: `v0.9.4...v0.9.5`

### Included

- The Neon Fang terminal banner already implemented in `install.sh`.
- The shorter, beginner-friendly README installation guidance.
- Release-documentation safety checks that keep release-specific notes
  historically accurate.
- Version synchronization across every application, daemon, package, lockfile,
  and installer source controlled by `app/scripts/version.mjs`.
- A new root changelog entry and a matching condensed in-app changelog entry.
- README pinned-checksum and manual-package examples updated to `v0.9.5`.
- Freshly built and verified desktop-app and daemon DEB/RPM packages.
- Short, beginner-friendly GitHub release notes describing only this release.

### Excluded

- New Fang application features.
- New hardware models or changes to hardware control.
- New Linux distribution, base-version, or architecture support.
- Changes to the immutable `v0.9.4` release.
- An installer-only release or an asset served directly from `main`.
- Package signing, package repositories, or automatic background updates.

## Desktop rebuild

Before pushing the release commit, rebuild Fang and `fangd` on this desktop
using the repository's production build and packaging paths. Rebuild both DEB
packages and both RPM packages, then run their existing metadata and lifecycle
verification where the host supports it.

The desktop rebuild is a preflight proof, not the source of the public release
assets. The tag-triggered GitHub workflow independently rebuilds and tests the
official packages from the exact tagged commit, so no locally produced binary
is manually uploaded or substituted into the immutable release.

## Exact release inventory

The release workflow must build, stage, validate, upload, and publish exactly
these six files:

1. `install.sh`
2. `SHA256SUMS`
3. `Fang_0.9.5_amd64.deb`
4. `fangd_0.9.5-1_amd64.deb`
5. `fang-0.9.5-1.x86_64.rpm`
6. `fangd-0.9.5-1.x86_64.rpm`

`SHA256SUMS` must describe the other five assets. The staged and remote
inventories must contain no additional files. Remote asset size and SHA-256
digest metadata must match the staged files before and after publication.

Although application and daemon behavior are unchanged, both package pairs are
rebuilt so their embedded versions and dependency bounds match `0.9.5`.

## Verification and publication sequence

The release is performed in this order:

1. Make the repository changes on `main`.
2. Rebuild the desktop application, daemon, both DEBs, and both RPMs on this
   desktop.
3. Run the complete local release gate, including version synchronization,
   version-script tests, installer fixtures, release-contract tests, publisher
   tests, DEB checks, RPM checks, Bash syntax, ShellCheck, and Git whitespace
   validation.
4. Confirm the worktree is clean and push `main`.
5. Wait for the exact pushed `main` commit's CI workflow to complete
   successfully. A green older commit is insufficient.
6. Confirm `v0.9.5` does not already exist as a local tag, remote tag, or GitHub
   release.
7. Confirm repository immutable releases are enabled and prepare the temporary
   publication credential described below.
8. Create and push tag `v0.9.5` at the exact green `main` commit.
9. Let `.github/workflows/release.yml` build both DEBs and both RPMs, test them
   on the declared Ubuntu, Debian, and Fedora matrix, stage the exact inventory,
   and publish one immutable release.
10. Wait for the tag-triggered release workflow to finish successfully.
11. Apply the approved title and body using GitHub's release-metadata edit
    operation. Do not upload, replace, rename, or remove any asset.
12. Verify the published release, live assets, installer behavior, and latest
    release resolution.
13. Delete the temporary repository secret after verification.

No tag is created before the repository changes, local gate, pushed-commit CI,
tag-absence check, immutable-setting check, and publication credential preflight
all pass.

## Publication credential

The publisher requires the repository secret
`IMMUTABLE_RELEASES_TOKEN` because it must read the repository's immutable
release setting through the GitHub API. The default workflow token remains the
credential used to create the draft, upload assets, and publish the release.

Before tagging:

- Confirm the authenticated credential can read the immutable-release setting
  and that the setting is enabled.
- Store that credential as the temporary `IMMUTABLE_RELEASES_TOKEN` repository
  secret without printing, logging, or writing its value into the repository.
- Confirm only that the secret name is present; do not attempt to read the
  secret back.

After the release and latest-installer verification succeed, delete
`IMMUTABLE_RELEASES_TOKEN` from the repository. Confirm the secret name is no
longer listed. Secret deletion is cleanup, not a reason to modify the
successfully published immutable release.

## Publisher behavior and failure handling

The existing publisher is the only approved publication path. It must:

- refuse to change a release that already exists for `v0.9.5`;
- create a draft only after the build and test jobs pass;
- upload exactly the six staged assets;
- validate the remote draft inventory, file sizes, and digests;
- publish once with `make_latest=true`;
- require the published release and the latest release to be immutable; and
- never modify `v0.9.4`.

If a pre-tag check fails, fix the repository or credential and repeat the
preflight without creating the tag.

If the release workflow fails before a draft exists, fix the cause and rerun
only when the publisher can still prove that no `v0.9.5` release exists. If a
draft was created, stop and inspect its exact state before taking action. Do not
blindly rerun publication, delete a published immutable release, replace
assets, or reuse a tag for different source. Any recovery must preserve the
publish-once contract.

## Post-publication acceptance criteria

The release is complete only when all of these are true:

- `v0.9.5` is published, stable, immutable, and reported as the latest release.
- Its target commit is the exact green `main` commit that was tagged.
- Its remote inventory is exactly the six filenames in this specification.
- Every remote asset digest and size matches the staged release inventory.
- `SHA256SUMS` verifies all five non-manifest assets.
- The DEB and RPM package metadata report `0.9.5` and matching package-pair
  dependency bounds.
- The attached `install.sh` passes Bash syntax and ShellCheck checks, is locked
  to `v0.9.5`, and contains the Neon Fang banner.
- Downloading
  `https://github.com/bladeandsoulx/fang-razer-linux/releases/latest/download/install.sh`
  returns the exact `v0.9.5` installer asset.
- Running the live installer far enough to render its non-destructive opening
  output displays the Neon Fang banner, not the old box banner.
- The GitHub release title, body, comparison link, and release flags match this
  specification.
- The temporary `IMMUTABLE_RELEASES_TOKEN` secret has been deleted.
- `v0.9.4` remains unchanged and immutable.
